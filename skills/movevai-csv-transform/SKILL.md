# MoveVAI CSV Transform

ใช้ skill นี้เมื่อโจทย์คืออ่านข้อมูลจาก CSV ต้นทาง แล้วแปลงเป็น format ปลายทางสำหรับการจัดส่งหรือส่งต่อระบบภายนอก

## หลักการทำงาน

- อย่ายึดชื่อไฟล์เป็น logic ของระบบ
- ให้ตัดสินจาก header, schema, และรูปแบบข้อมูลจริงของแต่ละคอลัมน์
- source อาจเป็น CSV ใดก็ได้ที่มีข้อมูลผู้ซื้อ, ที่อยู่, เบอร์โทร, จำนวน, และข้อมูลคำสั่งซื้อ
- target อาจเป็น schema จัดส่งที่ผู้ใช้ให้มาเป็นตัวอย่างหรือเป็นไฟล์อ้างอิง

## Workflow

1. อ่าน header ของ source และ target ก่อนเสมอ
2. สร้าง mapping จากความหมายของคอลัมน์ ไม่ใช่จากตำแหน่งคอลัมน์หรือชื่อไฟล์
3. แยกข้อมูลผู้ซื้อและผู้รับแบบ conservative
4. normalize เบอร์โทรและค่าปลายทางที่กำหนดตายตัว โดยคง raw address ตาม source เดิม
5. ถ้ามีเคสก้ำกึ่ง ให้เก็บค่า raw และส่งเข้าตรวจ ไม่เดาสุ่ม

## กฎสำคัญ

- `buyer_name` คือชื่อผู้ซื้อจากต้นทาง ห้าม overwrite
- `full_address` อาจมีชื่อผู้รับซ่อนอยู่ตอนต้น แม้ไม่มีคำว่า `ผู้รับ` หรือ `ส่งถึง`
- ถ้าใน `full_address` มีชื่อที่ต่างจาก `buyer_name` และอยู่ก่อน pattern ที่อยู่ไทยอย่างชัดเจน ให้ตีความว่าเป็นผู้รับ
- ถ้าไม่มั่นใจ ให้ใช้ผู้ซื้อเป็นค่าเริ่มต้นของผู้รับ และติดธงตรวจทาน
- ห้ามเติมหรือแต่งข้อมูลใหม่ที่ไม่มีอยู่จริงใน source
- ถ้า `full_address` ขัดกับ field แยก เช่น `Address` หรือ `Sub District` ห้ามผสมข้อมูลจนเกิดค่าที่ตรวจย้อนกลับไม่ได้
- เบอร์โทรไทยต้อง format ให้อ่านง่าย
- อย่าเติม quote (`'`) นำหน้าเบอร์โทร
- ถ้า target เป็น MoveVAI shipping CSV:
  `PRODUCT_IN_BOX` ต้องเป็นเลขลำดับแถวแบบ 1-based เสมอ
  `PRICE` ต้องเป็น `0` เสมอ
  `WEIGHT` ต้องเป็น `0` เสมอ
- ถ้า target เป็น MoveVAI shipping CSV ให้ส่ง `full_address` หรือ raw address field ที่ดีที่สุดไปที่ `RECEIVER_ADDRESS` ตรง ๆ
- ห้ามตัด, ย่อ, normalize, rewrite, หรือเรียบเรียง `RECEIVER_ADDRESS` ใหม่
- ถึงแม้ address จะมีหมู่บ้าน, อาคาร, ชั้น, ซอย, ถนน, หน้าร้าน, landmark, เขต/อำเภอ, จังหวัด, หรือรหัสไปรษณีย์ ปะปนอยู่ ให้คงไว้ตาม source เดิม
- ถ้า source มี `full_address` ให้ field นี้มีความสำคัญสูงสุดสำหรับ `RECEIVER_ADDRESS`

## การ format เบอร์โทร

- มือถือไทย 10 หลัก: `0XX-XXX-XXXX`
- เบอร์บ้านกรุงเทพ 9 หลักขึ้นต้น `02`: `02-XXX-XXXX`
- ถ้ารูปแบบไม่เข้าเงื่อนไข ให้คงเป็นตัวเลขที่ clean แล้ว

## การใช้ schema ปลายทางแบบขนส่ง

เมื่อปลายทางต้องการ field กลุ่ม `RECEIVER*` หรือ field จัดส่งลักษณะเดียวกัน ให้ใช้กฎ map ตาม reference:

- [references/mv-shipping-schema-th.md](references/mv-shipping-schema-th.md)

## สิ่งที่ต้องหลีกเลี่ยง

- อย่า hardcode ว่าต้นทางต้องชื่อ `examples.csv`
- อย่า hardcode ว่าปลายทางต้องชื่อ `mv_test.csv` หรือ `mv-01.csv`
- อย่าใช้ชื่อไฟล์เป็นตัวบอก schema
- อย่าเดาผู้รับอัตโนมัติทุกแถวโดยไม่มีระดับความมั่นใจ

## Output ที่คาดหวัง

- ไฟล์ปลายทางที่คง header ตาม schema ที่ผู้ใช้ให้
- ข้อมูลที่แปลงแล้วอ่านง่ายและพร้อมใช้งาน
- ถ้าจำเป็น ให้สรุป mapping และ assumption ที่ใช้แบบสั้น
